# IGESS v0.6 Snapshot Query and Profile Batch Design

## Context

IGESS v0.5 established the Agent Analyst workflow. Agents can run simulations,
review existing runs, compare against baselines, generate reports, and produce
advice artifacts without taking ownership of bulk Luban table edits.

v0.6 should extend that Agent workflow with stronger simulation observability.
The approved scope is intentionally narrow:

- Agents can ask for an exact player-state snapshot at a specific scenario,
  profile, and time.
- Agents can run the profiles already declared in a YAML scenario as separate
  batch lanes and receive a manifest that indexes the outputs.

v0.6 does not add temporary profile variants, snapshot resume/fork simulation,
or automatic table editing.

## Approaches Considered

1. **Snapshot Query plus Existing Profile Batch**
   - Adds an exact `snapshot` command and a profile-oriented `batch-run`
     command.
   - Uses YAML-defined profiles only.
   - Keeps the current deterministic simulator and output contract intact.
   - Recommended for v0.6.

2. **Snapshot Query plus Automatic Key-Event Snapshots**
   - Adds the exact snapshot command and also writes snapshots around first
     purchases, unlocks, prestige resets, and bottleneck windows.
   - Stronger report evidence, but it requires a new rule set for deciding which
     events deserve snapshots.
   - Better as a follow-up once manual snapshot queries are stable.

3. **Full Profile Matrix with Profile Variants**
   - Adds existing profile batch runs plus temporary profile overrides such as
     efficiency and policy changes.
   - Powerful for experiments, but it broadens the scope into parameter matrix
     design and risks overlapping with the existing scan feature.
   - Deferred beyond v0.6.

## Scope

v0.6 includes:

- `igess snapshot`.
- Exact simulation to a requested `time_seconds`, including non-record-interval
  times such as `73`.
- Snapshot output as `snapshot.json` and `snapshot.md`.
- A snapshot payload richer than `TimelineRow`, including unlocked content,
  milestone state, prestige counts, recent event context, and the next
  recommended action when available.
- `igess batch-run`.
- Batch runs for the profiles already declared by the selected scenario.
- One isolated run directory per profile lane.
- `batch_manifest.json` and `batch_summary.json`.
- Deterministic output ordering and LF newlines.
- Clear errors for invalid scenario, profile, time, or missing output paths.

v0.6 defers:

- Profile variants or inline profile override expressions.
- Snapshot resume/fork simulation.
- Persisted save-game import/export.
- Automatic key-event snapshot selection for reports.
- Parallel execution as a correctness requirement. The batch command may run
  lanes sequentially first, as long as the output contract can support later
  parallelization.

## Commands

### Snapshot

```powershell
igess snapshot --config examples/shelldiver_v0/economy.yaml --tables examples/shelldiver_v0/luban_exports --scenario day_1_progression --profile optimizer --time 73 --out .tmp/snapshot
```

Outputs:

- `.tmp/snapshot/snapshot.json`
- `.tmp/snapshot/snapshot.md`

The command validates config and table exports before simulation. It rejects:

- Unknown scenario ids.
- Profiles not included in the requested scenario.
- Negative times.
- Times greater than scenario duration.
- Non-integer time values.

### Batch Run

```powershell
igess batch-run --config examples/shelldiver_v0/economy.yaml --tables examples/shelldiver_v0/luban_exports --scenario day_1_progression --out .tmp/batch
```

Outputs:

- `.tmp/batch/batch_manifest.json`
- `.tmp/batch/batch_summary.json`
- `.tmp/batch/profiles/<profile_id>/run_manifest.json`
- `.tmp/batch/profiles/<profile_id>/timeline.json`
- `.tmp/batch/profiles/<profile_id>/events.json`
- Existing run artifacts for each profile lane.

The command uses the scenario's declared profile list. It does not create new
profiles or change profile configuration.

## Snapshot Contract

`snapshot.json` should use schema version 1:

```json
{
  "schema_version": 1,
  "scenario_id": "day_1_progression",
  "profile_id": "optimizer",
  "time_seconds": 73,
  "model_id": "shelldiver_incremental_v0",
  "resources": {
    "fish": "12.34"
  },
  "generators_owned": {
    "fisherman": 1
  },
  "upgrades_purchased": [],
  "unlocked_generators": ["fisherman"],
  "unlocked_upgrades": [],
  "milestones_claimed": [],
  "prestige_counts": {},
  "total_cps": "1.05",
  "recent_events": [],
  "next_recommended_action": {
    "kind": "buy_generator",
    "item_id": "fisherman",
    "cost_resource": "fish",
    "cost": "11.5",
    "score": "0.09"
  }
}
```

`snapshot.md` is a short Agent-readable explanation:

- Header with scenario, profile, and time.
- Current resources and total CPS.
- Owned generators and purchased upgrades.
- Unlock and prestige state.
- Recent events.
- Next recommended action or a clear reason when none is available.

## Batch Contract

`batch_manifest.json` should use schema version 1:

```json
{
  "schema_version": 1,
  "scenario_id": "day_1_progression",
  "model_id": "shelldiver_incremental_v0",
  "profiles": ["casual", "optimizer", "explorer"],
  "lanes": [
    {
      "profile_id": "casual",
      "status": "success",
      "run_dir": "profiles/casual",
      "artifacts": [
        "run_manifest.json",
        "timeline.json",
        "events.json",
        "analysis.json",
        "analysis.md",
        "payback.csv"
      ]
    }
  ]
}
```

`batch_summary.json` should include compact final metrics by profile:

- Final resources.
- Final total CPS.
- Purchase count.
- Unlock count.
- Prestige count.
- First and last event times when available.

## Architecture

### `snapshot.py`

New module responsible for snapshot payloads and rendering:

- `SnapshotService.snapshot_at(model, scenario_id, profile_id, time_seconds)`.
- `SnapshotService.write(snapshot, output_dir)`.
- Markdown rendering for `snapshot.md`.

The service should not load config files directly. CLI and workflow callers load,
lint, and build the model before invoking it.

### `simulator.py`

Add a narrow API:

```python
Simulator.snapshot_at(scenario_id: str, profile_id: str, time_seconds: int)
```

This method simulates only the requested profile up to the exact requested time.
It should share the same state transition code as `run_scenario` so snapshot
results remain consistent with full runs.

The implementation should avoid a forked simulator. A private helper may run one
profile lane:

```python
_run_profile_until(scenario, profile_id, end_time, record_interval_seconds=None)
```

`run_scenario` and `snapshot_at` can both use that helper.

### `batch.py`

New module responsible for profile batch orchestration:

- Validate that the scenario exists.
- Iterate over the scenario's existing profiles.
- For each profile, run only that profile lane and write normal run artifacts.
- Write `batch_manifest.json`.
- Write `batch_summary.json`.

The first implementation may run lanes sequentially. The manifest shape should
not prevent later use of `concurrent.futures` for true local parallel execution.

### `cli.py`

Add two commands:

- `snapshot`
- `batch-run`

`cli.py` should remain a thin coordinator. It should not build snapshot or batch
payloads inline.

### Existing Outputs

`OutputWriter.write_all` should continue writing the existing artifact set for
normal runs. Batch lanes may reuse this writer for each profile-specific result.
The batch command writes its own batch-level manifest beside the lane outputs.

## Data Flow

Snapshot flow:

1. CLI receives config, tables, scenario, profile, time, and output directory.
2. `ConfigLoader` loads YAML and table exports.
3. `ConfigLinter` validates raw config.
4. `ModelBuilder` builds the economy model.
5. `SnapshotService` validates scenario/profile/time.
6. `Simulator.snapshot_at` runs the profile lane to the exact time.
7. `SnapshotService` builds `snapshot.json` and `snapshot.md`.

Batch flow:

1. CLI receives config, tables, scenario, and output directory.
2. `ConfigLoader`, `ConfigLinter`, and `ModelBuilder` prepare the model.
3. `BatchRunner` reads the scenario's declared profiles.
4. Each profile lane is simulated independently.
5. `OutputWriter.write_all` writes normal artifacts under
   `profiles/<profile_id>`.
6. `BatchRunner` writes the batch manifest and summary.

## Error Handling

Errors should be explicit and actionable:

- `scenario 'x' not found`
- `profile 'x' is not part of scenario 'day_1_progression'`
- `snapshot time must be >= 0`
- `snapshot time 9999 exceeds scenario duration 180`
- `batch scenario 'x' has no profiles`

CLI commands return exit code `1` for validation failures, matching existing
IGESS command behavior.

Partial batch failures should be recorded per lane in `batch_manifest.json` if
lane-level isolation is implemented. The first v0.6 implementation can fail the
whole batch on the first lane error because all lanes share the same validated
model.

## Testing

Add focused tests:

- Snapshot output is deterministic across repeated calls.
- Snapshot at a non-record-interval time returns exactly that time.
- Snapshot rejects unknown profile, profile outside the scenario, negative time,
  and time beyond duration.
- Snapshot includes richer state than `TimelineRow`, including unlock,
  milestone, and prestige fields.
- Snapshot next recommended action matches the policy engine for that state.
- `batch-run` writes one lane directory per scenario profile.
- `batch_manifest.json` indexes every lane with stable relative paths.
- `batch_summary.json` includes compact final metrics per profile.
- Existing v0.5 commands still pass their smoke tests.

Full verification target:

```powershell
python -m pytest
python -m igess.cli snapshot --config examples/shelldiver_v0/economy.yaml --tables examples/shelldiver_v0/luban_exports --scenario day_1_progression --profile optimizer --time 73 --out .tmp/snapshot
python -m igess.cli batch-run --config examples/shelldiver_v0/economy.yaml --tables examples/shelldiver_v0/luban_exports --scenario day_1_progression --out .tmp/batch
```

## Acceptance Criteria

v0.6 is complete when:

- `igess snapshot` produces deterministic JSON and Markdown for an exact
  scenario/profile/time.
- Snapshot supports non-record-interval times.
- Snapshot output includes resources, owned generators, purchased upgrades,
  unlocked content, milestone state, prestige state, total CPS, recent events,
  and next recommended action.
- Invalid snapshot requests fail with clear messages.
- `igess batch-run` writes an indexed batch directory using the profiles already
  declared in YAML.
- Batch lane outputs can be consumed by the existing report and review tooling.
- Existing run, report, compare, gate, advise, and review-run behavior remains
  compatible.
