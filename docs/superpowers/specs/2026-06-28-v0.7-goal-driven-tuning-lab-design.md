# IGESS v0.7 Goal-Driven Tuning Lab Design

## Context

IGESS v0.4 established the local tuning workflow: export, lint, run, report,
compare, scan, gate, doctor, and explain.

IGESS v0.5 added the Agent Analyst workflow. Agents can run deterministic
analysis, review artifacts, compare against baselines, and produce advice while
humans keep ownership of bulk Luban table edits.

IGESS v0.6 is the observability layer: exact scenario/profile/time snapshots and
YAML-defined profile batch runs. It gives agents and designers stronger evidence
for where progression stalls and how different player profiles diverge.

v0.7 should turn that evidence into an actionable tuning loop. The tool should
answer the designer's next question:

> Which safe table changes best move this economy toward my targets, and why?

The v0.7 theme is **Goal-Driven Tuning Lab**.

## Product Principle

v0.7 should make IGESS a recommendation engine for bounded, auditable tuning
experiments. It should not become an automatic data-table editor.

The intended ownership boundary remains:

- **Human**: edits Luban/Excel source tables and decides whether to apply
  recommended value changes.
- **Agent**: runs deterministic experiments, ranks candidates, explains results,
  and produces human-readable table recommendations.
- **IGESS**: provides reproducible artifacts, stable schemas, and evidence that
  make each recommendation inspectable.

## Approaches Considered

1. **Goal-Driven Tuning Lab**
   - Adds `igess tune`.
   - Lets YAML define tuning targets and candidate parameter ranges.
   - Runs bounded experiments, evaluates them against gates and targets, ranks
     the best candidates, and writes evidence-rich recommendation artifacts.
   - Recommended for v0.7 because it composes existing scan, compare, gate,
     advice, snapshot, and batch-run capabilities into a higher-value workflow.

2. **New Economy Mechanics**
   - Adds deferred mechanics such as probability gates, build delays, complex
     offline caps, and deeper multi-layer prestige balancing.
   - Useful, but mainly expands simulator realism. It does not directly improve
     the designer's ability to decide which current values to change.
   - Better after the tuning loop can measure and rank changes.

3. **Visual Data Table Editor**
   - Adds a dashboard UI for editing Luban/Excel-style values.
   - Potentially convenient, but it conflicts with the v0.5 authorship boundary
     if agents begin applying bulk source-table changes.
   - Better deferred until IGESS has stronger review/approval mechanics around
     human-owned table edits.

## Scope

v0.7 includes:

- `igess tune`.
- YAML tuning target definitions.
- YAML tuning candidate definitions for bounded table-field ranges.
- Single-parameter and small multi-parameter candidate generation.
- Bounded candidate count with explicit failure when a search expands too much.
- Scenario/profile evaluation using deterministic simulation artifacts.
- Target scoring with profile-aware weights.
- Optional baseline comparison when a baseline run or batch is supplied.
- Optional gate evaluation when regression gates are configured.
- Ranked candidate output in JSON and Markdown.
- Human-only table recommendations that never edit Luban/Excel files.
- Dashboard display of the latest tuning report.

v0.7 defers:

- Automatic Excel/Luban source-table writes.
- Search algorithms beyond small deterministic grid search.
- Bayesian optimization, genetic search, or stochastic sampling.
- Cloud execution or hosted tuning jobs.
- New economy mechanics such as probability gates and build delays.
- Full visual editing of tuning candidate spaces.

## Commands

### Tune

```powershell
igess tune --config examples/shelldiver_v0/economy.yaml --tables examples/shelldiver_v0/luban_exports --scenario day_1_progression --out .tmp/tune
```

Outputs:

- `.tmp/tune/tuning_report.json`
- `.tmp/tune/tuning_report.md`
- `.tmp/tune/candidates/<candidate_id>/run_manifest.json`
- Existing run artifacts for each evaluated candidate.

Optional baseline:

```powershell
igess tune --config examples/shelldiver_v0/economy.yaml --tables examples/shelldiver_v0/luban_exports --scenario day_1_progression --baseline .tmp/batch --out .tmp/tune
```

The baseline may be a normal run directory or a v0.6 batch directory. When a
baseline exists, tuning reports include deltas against it.

## YAML Contract

v0.7 adds two optional sections to the economy YAML.

### `tuning_targets`

Example:

```yaml
tuning_targets:
  day_1_progression:
    profile_weights:
      casual: 1.5
      optimizer: 1.0
      explorer: 1.0
    metrics:
      max_purchase_gap_seconds:
        casual: 60
      max_unlock_time_seconds:
        generator:fisherman:
          casual: 30
          optimizer: 20
      min_prestige_gain:
        optimizer: 1
```

Supported target metrics in v0.7:

- `max_purchase_gap_seconds`
- `max_unlock_time_seconds`
- `max_payback_seconds`
- `min_prestige_gain`
- `max_prestige_time_seconds`

The target set should intentionally overlap with existing analysis and gate
metrics. v0.7 should not invent broad new metric families unless existing run
artifacts cannot support a required designer question.

### `tuning_candidates`

Example:

```yaml
tuning_candidates:
  day_1_progression:
    max_candidates: 120
    fields:
      - table: generators
        row_id: fisherman
        field: cost_growth
        values: [1.11, 1.12, 1.13, 1.14]
      - table: generators
        row_id: boat
        field: base_cost
        range:
          start: 95
          end: 115
          step: 5
```

Candidate fields identify exported table data, not source workbook cells. The
report maps recommendations back to workbook names through existing source
metadata whenever possible.

v0.7 supports:

- Explicit `values`.
- Numeric `range` with inclusive `start`, `end`, and positive `step`.
- `max_candidates` required or defaulted to a conservative cap.
- Deterministic candidate IDs.
- At most a small grid product. If the product exceeds `max_candidates`, the
  command fails with a clear message.

v0.7 does not support:

- Editing YAML during tuning.
- Applying recommendations to source workbooks.
- String, boolean, object, or formula-expression search values.
- Unknown table fields or missing row IDs.

## Tuning Report Contract

`tuning_report.json` should use schema version 1:

```json
{
  "schema_version": 1,
  "scenario_id": "day_1_progression",
  "model_id": "shelldiver_incremental_v0",
  "status": "candidates_found",
  "summary": "3 candidates improved weighted target score.",
  "baseline": {
    "path": ".tmp/batch",
    "kind": "batch"
  },
  "targets": {},
  "candidate_count": 24,
  "best_candidates": [
    {
      "candidate_id": "cand_0001",
      "rank": 1,
      "score": "0.82",
      "status": "recommended",
      "changes": [
        {
          "table": "generators",
          "workbook": "generators.xlsx",
          "row_id": "fisherman",
          "field": "cost_growth",
          "current_value": "1.15",
          "suggested_value": "1.13",
          "apply_mode": "human_only"
        }
      ],
      "target_results": [],
      "profile_summary": {},
      "evidence": {
        "run_dir": "candidates/cand_0001",
        "report_dir": null,
        "snapshot_refs": []
      },
      "warnings": []
    }
  ],
  "rejected_candidates": []
}
```

`tuning_report.md` should be designer-readable:

- Header with scenario, candidate count, and status.
- Summary of whether any candidate improves the targets.
- Top recommended candidates.
- Exact human table edits for each candidate.
- Per-profile metric changes.
- Gate pass/fail notes when available.
- Evidence paths for run artifacts and reports.
- Warnings for trade-offs, regressions, or weak improvements.

## Scoring

The first scoring model should be deterministic and easy to explain.

For each candidate:

1. Run the scenario or batch lanes.
2. Extract supported metrics from run artifacts.
3. Convert each target into a normalized pass or distance-to-target score.
4. Apply profile weights.
5. Penalize gate failures and invalid-content findings.
6. Sort candidates by score descending, then by smaller change magnitude, then by
   stable candidate ID.

The exact scoring constants should live in one focused module so future versions
can tune them without changing command orchestration.

## Architecture

### `tuning.py`

New service module responsible for:

- Parsing and validating tuning target/candidate config from raw YAML.
- Generating candidate override sets.
- Running deterministic candidate simulations.
- Scoring and ranking candidate results.
- Writing `tuning_report.json` and `tuning_report.md`.

Suggested public API:

```python
def run_tuning(
    config: str | Path,
    tables: str | Path,
    scenario_id: str,
    output_dir: str | Path,
    baseline: str | Path | None = None,
) -> Path:
    ...
```

### `tuning_config.py`

Optional focused module if `tuning.py` grows too large. It owns dataclasses and
validation helpers for `tuning_targets` and `tuning_candidates`.

### `metrics.py`

Extend only if current metrics do not expose required target values. Prefer
small helpers that reuse `timeline.json`, `events.json`, `analysis.json`, and
`payback.csv`.

### `scan.py`

Reuse range parsing behavior where practical, but avoid forcing all tuning
candidate logic through the existing single-parameter scan command. `scan`
remains a low-level experiment tool; `tune` is the goal-driven orchestration
layer.

### `advice.py`

The Agent Analyst may link to tuning reports or summarize the best candidate in
a later version. v0.7 does not need to merge advice and tuning artifacts.

### `dashboard.py` and `workflows.py`

Add a small dashboard panel for the latest tuning report:

- Run Tuning button.
- Latest tuning status.
- Top candidate table edits.
- Link to candidate artifacts when available.

Keep dashboard actions as deterministic wrappers around local IGESS commands.

### `cli.py`

Add a thin `tune` command branch. It should not contain scoring or candidate
generation logic inline.

## Data Flow

1. CLI receives config, tables, scenario, optional baseline, and output path.
2. `ConfigLoader` loads YAML and table exports.
3. `ConfigLinter` validates existing economy config plus tuning sections.
4. `ModelBuilder` builds the base model.
5. `TuningService` validates targets and candidate ranges for the scenario.
6. Candidate overrides are generated deterministically.
7. Each candidate is simulated into an isolated output directory.
8. Candidate metrics are extracted from artifacts.
9. Scores are calculated and ranked.
10. `tuning_report.json` and `tuning_report.md` are written.
11. Dashboard and Agent workflows read the same report artifacts.

## Error Handling

Errors should be explicit and actionable:

- `scenario 'x' not found`
- `scenario 'x' has no tuning_targets`
- `scenario 'x' has no tuning_candidates`
- `tuning candidate field generators.fisherman.cost_growth not found`
- `tuning candidate field generators.fisherman.cost_growth is not numeric`
- `tuning range step must be > 0`
- `tuning candidate grid expands to 240 variants, over max_candidates 120`
- `tuning target metric 'x' is not supported`
- `baseline path does not contain a readable run or batch manifest`

Partial candidate failures should be captured in `tuning_report.json` when the
base config is valid and only a candidate fails. The command should return exit
code `1` only when no valid report can be produced or when config validation
fails before candidate evaluation.

## Safety Rules

- `igess tune` must never write to `data-tables/Datas`.
- `igess tune` must never modify `.xlsx` files.
- `igess tune` must never apply YAML changes.
- Recommendations must use `apply_mode: human_only`.
- Every recommended change must cite current value, suggested value, table, row,
  field, and evidence path.
- Candidate counts must be capped.
- All artifact paths in reports must be relative to the tuning output directory
  where practical.

## Testing

Add focused tests:

- Tuning config validation accepts a valid target/candidate section.
- Tuning config validation rejects unknown metrics.
- Candidate range expansion is deterministic.
- Candidate expansion rejects grids larger than `max_candidates`.
- `igess tune` writes JSON and Markdown reports for the sample project.
- Tuning report recommendations never modify workbook/source table files.
- Candidate ranking is stable across repeated runs.
- Baseline comparison changes report output when a baseline is supplied.
- Dashboard can render latest tuning status.
- Existing `scan`, `gate`, `advise`, `run`, `report`, and `compare` tests remain
  compatible.

Full verification target:

```powershell
python -m pytest
python -m igess.cli tune --config examples/shelldiver_v0/economy.yaml --tables examples/shelldiver_v0/luban_exports --scenario day_1_progression --out .tmp/tune
```

## Acceptance Criteria

v0.7 is complete when:

- `igess tune` produces deterministic JSON and Markdown reports.
- YAML can define profile-weighted tuning targets.
- YAML can define bounded numeric candidate spaces.
- Candidate evaluation writes isolated artifacts.
- Candidate ranking is stable and explainable.
- Top candidates include exact human-only table recommendations.
- Reports include per-profile evidence and warnings.
- Candidate count limits prevent accidental large searches.
- Source workbooks and table files are not modified by tuning commands.
- Dashboard exposes the latest tuning report without becoming a table editor.
- Existing v0.4-v0.6 workflows remain compatible.
