# IGESS v0.9 Web Reporting Design

## Context

IGESS already writes deterministic simulation artifacts:

- `timeline.json` / `timeline.csv`
- `events.json` / `events.csv`
- `analysis.json` / `analysis.md`
- `payback.csv`
- `run_manifest.json`

The current `igess report` command turns those artifacts into a self-contained
HTML report with one lightweight SVG resource curve, event and payback tables,
analysis warnings, and trace lists. The local dashboard can run scenarios and
serve those static reports from run history.

That is enough for first-pass inspection, but not enough for the v0.9 product
goal: a designer should open a Web report and quickly understand progression
rhythm, player-profile divergence, major stalls, and the source evidence behind
those stalls.

v0.9 should therefore focus on **shared Web reporting capability** for both
static reports and the dashboard. It should not wait for the future tuning loop,
but it should leave clean extension points for tuning and baseline comparison
views.

## Goal

Build a shared Web report layer that helps designers answer two questions:

1. Is the progression rhythm healthy across profiles?
2. If not, where is it stuck and what evidence explains the problem?

The report should preserve IGESS's deterministic, local-first workflow while
allowing a stronger front-end charting library.

## Recommended Direction

Use an **ECharts-backed shared report shell**.

The report shell should be consumed by:

- `igess report`, which writes an offline-friendly static report folder.
- `igess dashboard`, which serves the same report shell and data model for run
  history entries.

The first implementation should avoid a full React/Vite application. IGESS can
get most of the value with generated HTML, a shared CSS file, a modular report
JavaScript file, and a vendored `echarts.min.js` asset. That keeps packaging
simple, preserves direct report-folder sharing, and avoids splitting dashboard
and static report rendering logic.

## Approaches Considered

### 1. Shared ECharts Report Shell

This adds a normalized report data model and a reusable browser shell powered by
Apache ECharts.

Pros:

- Supports resource curves, CPS curves, timeline/event charts, bar charts,
  tooltips, legends, and zoom with less custom JavaScript.
- Allows static reports and dashboard reports to share one rendering path.
- Keeps report output inspectable and local-first.
- Leaves a clean path for future tuning and comparison visualizations.

Cons:

- Adds a front-end library asset to the package.
- Requires tests that verify generated HTML references the correct assets and
  data payload.

This is the recommended v0.9 direction.

### 2. Dashboard-Only Rich Reporting

This puts richer interactive reports only in the local dashboard and leaves
static `igess report` mostly unchanged.

Pros:

- Easier to build a more app-like interaction model.
- Less concern about fully self-contained static output.

Cons:

- Static reports remain weak.
- Dashboard and report rendering logic would diverge.
- Sharing and archiving analysis results would still be table-heavy.

This is not recommended for v0.9.

### 3. Keep Custom SVG Charts

This expands the current hand-written SVG chart implementation.

Pros:

- No new front-end dependency.
- Very small generated report assets.

Cons:

- Tooltips, zoom, legends, event lanes, responsive sizing, and bar charts become
  repetitive custom work.
- The custom chart layer would likely become a maintenance burden before tuning
  views arrive.

This is not recommended now that front-end library usage is allowed.

## Scope

v0.9 includes:

- Shared report data package generation.
- Shared report shell assets for static report and dashboard report serving.
- Apache ECharts as the charting engine.
- Resource curve chart.
- CPS curve chart.
- Event timeline or event lane chart.
- Payback pressure bar chart.
- Diagnostic summary panel for bottlenecks, invalid content, overpowered
  content, never-purchased content, and never-unlocked content.
- Linked evidence details for selected chart/table items where current artifacts
  expose source references or formula traces.
- Responsive report layout for normal desktop and narrow browser widths.
- Test coverage for data model generation, report output, dashboard serving, and
  basic HTML/data contracts.

v0.9 defers:

- Full tuning candidate ranking views.
- Deep baseline-vs-candidate charting.
- PNG/PDF export.
- React/Vite or a long-lived front-end application build pipeline.
- Editing values from the report UI.
- Automatic source workbook writes.

## Report Data Model

Add a stable browser-facing report payload, written as `report_data.json` in
static report output and loaded in the dashboard from the same service layer.

Suggested top-level shape:

```json
{
  "schema_version": 1,
  "scenario": {
    "id": "day_1_progression",
    "model_id": "shelldiver_incremental_v0",
    "profiles": ["casual", "explorer", "optimizer"]
  },
  "overview": {},
  "series": {
    "resources": [],
    "total_cps": [],
    "events": []
  },
  "diagnostics": {
    "bottlenecks": [],
    "invalid_content": {},
    "overpowered_content": [],
    "payback": []
  },
  "evidence": {
    "traces": [],
    "source_refs": []
  },
  "artifacts": {}
}
```

The browser payload should be derived from existing `ReportData` instead of
forcing the browser to understand raw run artifacts directly.

The report generator should also embed the same payload in `index.html` inside a
`<script type="application/json">` tag. Many browsers block `fetch()` for local
`file://` pages, so direct-open static reports need an inline fallback. The
long-term machine-readable contract remains `report_data.json`; the inline copy
is a browser-delivery convenience for archive-friendly static reports.

### Numeric Values

IGESS economy values may be normal decimals, very large number strings, or
special values such as `Infinity`. The browser payload should not assume every
display value can be safely converted with JavaScript `Number()`.

For charted values, the view model should emit both:

- `display_value`: the exact string or formatted value shown to the designer.
- `chart_value`: a finite JavaScript-safe number used only for plotting.

When a value cannot be represented safely, the view model should either:

- provide a logarithmic or normalized chart value with clear axis labeling, or
- exclude it from the chart and surface it in the relevant warning list.

This avoids losing idle-game scale information while still allowing ECharts to
render stable charts.

## Report Views

### Overview

Show scenario id, model id, profile count, duration coverage, timeline row count,
event count, and missing optional artifacts.

Purpose:

- Confirm the report was generated from the expected run.
- Surface malformed or partial runs early.

### Resource Curves

Use ECharts line charts to show resource value over time.

Required behavior:

- Resource selector.
- One series per profile.
- Tooltip with time, profile, and value.
- Legend toggles by profile.
- Data zoom for longer timelines.

### CPS Curves

Show `total_cps` over time by profile.

Purpose:

- Make growth-rate changes visible.
- Reveal whether a purchase, milestone, upgrade, or prestige event actually
  changes production speed.

### Event Timeline

Show purchase, unlock, milestone, prestige, and offline events on a time axis.

Acceptable first implementation:

- Scatter or lane chart grouped by profile.
- Event kind shown by symbol or color.
- Tooltip shows profile, time, kind, item id, and selected details.

### Payback Pressure

Show payback seconds as a horizontal bar chart.

Required behavior:

- Group or filter by profile.
- Sort by longest finite payback first.
- Show `Infinity` entries in a separate warning list or capped visual bucket.
- Tooltip includes source workbook/table/row where available.

### Diagnostics

Show high-signal warnings from `analysis.json`:

- Never purchased.
- Never unlocked.
- Overpowered content.
- Bottleneck gap counts by profile.
- Worst payback entries.

Each diagnostic item should point to available evidence:

- source workbook/table/row
- formula trace
- event id or time
- artifact name

### Evidence Details

Keep formula traces and source references out of the main visual flow. Show them
in a details area, side panel, or expandable section.

The first version can use normal HTML details elements. The key requirement is
that a designer can move from "this chart looks wrong" to "which row/formula
produced this?" without opening raw JSON manually.

## Architecture

### `reporting/view_model.py`

New module that converts `ReportData` into the browser-facing report payload.

Responsibilities:

- Normalize timeline rows into chart-ready series.
- Extract resource ids, profile ids, event kinds, payback rows, diagnostics, and
  trace evidence.
- Preserve artifact/source references.
- Avoid formatting decisions that belong in the browser.

### `reporting/assets/`

New package directory for shared browser assets:

- `report.css`
- `report.js`
- `echarts.min.js`

The ECharts asset should be vendored or installed in a deterministic way that
does not require internet access when generating reports.

### `reporting/static.py`

Update static report generation to:

- Build the view model.
- Write `report_data.json`.
- Copy shared assets.
- Render a lightweight HTML shell.

It should no longer own large inline CSS/JS strings once the shared assets exist.

### `dashboard.py`

Update dashboard report serving so dashboard run history opens the same report
shell that static report generation writes.

The dashboard home can remain simple in v0.9. Its main responsibility is to
launch deterministic workflows and link to report pages.

### `workflows.py`

Keep workflow orchestration unchanged where possible. It should still produce
normal run artifacts and call report generation.

## Data Flow

1. A run writes normal IGESS artifacts.
2. `load_report_data(run_dir)` reads those artifacts.
3. `build_report_view_model(report_data)` creates the browser payload.
4. Static report generation writes:
   - `index.html`
   - `report_data.json`
   - `assets/report.css`
   - `assets/report.js`
   - `assets/echarts.min.js`
5. Browser reads the inline JSON payload when present, otherwise fetches
   `report_data.json`.
6. Dashboard serves the same report folder for each run history entry.

## Error Handling

Report generation should fail loudly for missing required artifacts that
`load_report_data` already treats as required.

Optional artifact gaps should be visible in the report:

- Missing `payback.csv` should hide or disable the payback chart and show a clear
  note.
- Missing trace data should show "No trace evidence available" rather than an
  empty panel.
- Empty timelines should render a clear empty state.

The browser should not throw uncaught errors when a chart has no data.

## Packaging

ECharts should be available without network access during report viewing.

Acceptable packaging options:

- Commit a known `echarts.min.js` asset under `src/igess/reporting/assets/`.
- Or add an explicit Python package data step that installs and copies the asset
  from a pinned source.

The first option is simpler for v0.9.

## Testing

Add focused tests:

- View model generation includes scenario, profiles, resource series, CPS series,
  events, diagnostics, payback rows, and evidence fields for the sample run.
- `generate_static_report` writes `report_data.json` and shared assets.
- Generated HTML references `assets/echarts.min.js`, `assets/report.js`, and
  `report_data.json`.
- Reports handle missing optional `payback.csv`.
- Dashboard run history still serves report files and blocks path traversal.
- Existing `report`, `dashboard`, `advise`, `verify-edits`, and `compare` tests
  remain compatible.

Manual verification:

```powershell
.\.venv\Scripts\python -m pytest
.\.venv\Scripts\python -m igess.cli run --config examples/shelldiver_v0/economy.yaml --tables examples/shelldiver_v0/luban_exports --scenario day_1_progression --out .tmp/report_run
.\.venv\Scripts\python -m igess.cli report --run .tmp/report_run --out .tmp/web_report --title "IGESS v0.9 Report"
```

Then open `.tmp/web_report/index.html` and verify:

- Resource chart renders.
- CPS chart renders.
- Event timeline renders.
- Payback pressure view renders or shows a clear empty state.
- Diagnostic panel shows current warnings.
- No browser console errors appear for the sample run.

## Acceptance Criteria

v0.9 is complete when:

- Static reports and dashboard reports share one report data model and browser
  shell.
- `igess report` produces a Web report with ECharts-backed resource, CPS, event,
  and payback views.
- Reports show diagnostic summaries with links or references to available
  evidence.
- Report output remains local and archive-friendly.
- Dashboard run history continues to serve generated reports.
- Missing optional artifacts produce clear empty states.
- Tests cover the report data contract and generated file layout.
- Existing command-line workflows remain compatible.

## Non-Goals

- Editing economy values in the report.
- Applying recommendations from the report.
- Replacing Luban/Excel source workflows.
- Hosted report sharing.
- Cloud dashboards.
- Full tuning candidate visualization before the tuning workflow exists.
